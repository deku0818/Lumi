# Prompt Caching 参考

> 本文整合了 Anthropic Prompt Caching 的实战教程和 Claude Code 团队的工程经验，帮助理解缓存原理并在 Agent 项目中正确使用。

---

## 核心原理：前缀匹配

Prompt 缓存基于**严格的前缀匹配**：从第一个字节开始逐字匹配，一旦某处不同，该处之后的所有内容全部缓存失效。

```
请求:  [A] [B] [C] [D] [E（新）]
缓存:  [A] [B] [C] [D]
                          ↑ 从这里开始才需要计算
```

**改前面一个字，后面全部白算。**

---

## 请求结构最佳排列

原则：**越稳定的内容越靠前，越容易变的内容越靠后。**

```
┌─────────────────────────────────┐
│  ① 静态系统提示词 & 工具定义     │  ← 全局不变，缓存断点
├─────────────────────────────────┤
│  ② 项目配置（如 CLAUDE.md）      │  ← 项目级不变，缓存断点
├─────────────────────────────────┤
│  ③ 对话历史消息                  │  ← 每轮追加，无需标记
├─────────────────────────────────┤
│  ④ 最新一轮消息                  │  ← 缓存断点
└─────────────────────────────────┘
```

---

## 缓存断点标记策略

`cache_control: {"type": "ephemeral"}` 是缓存断点标记。采用**滑动窗口**策略：

- 只标在 system prompt 和对话末尾最后 1-2 条消息
- 中间消息无需标记 — 前面不变时自动被缓存前缀覆盖
- 每轮新请求，断点随对话"向后滑动"

---

## 动态信息处理

**不要**把时间戳等动态信息写在系统提示词里（每次变动 → 缓存全部失效）。

**正确做法**：通过对话消息中的 `<system-reminder>` 标签传递更新，保持系统提示词不变。

---

## 关键禁忌

### 不要中途切换模型

缓存按模型隔离。10 万 token 的 Opus 对话切到 Haiku 需要**从零重建缓存**。正确做法：用**子 Agent** 交接。

### 不要中途增减工具

工具定义是缓存前缀的一部分。增减任何工具 → 缓存全部失效。

**替代方案 1**：用工具本身建模状态切换（如 `EnterPlanMode`/`ExitPlanMode`）

**替代方案 2**：延迟加载（`defer_loading`），发送轻量级桩 + `ToolSearch` 按需加载

### 上下文压缩要保缓存

压缩请求必须使用与主对话**完全相同**的系统提示词、工具定义和历史消息，只在末尾追加压缩指令，复用父对话缓存。

---

## 速查表

| 原则 | 做法 | 原因 |
|------|------|------|
| 内容排列 | 静态在前，动态在后 | 前缀匹配 |
| 缓存断点 | 只标在 system prompt 和对话末尾 | 滑动窗口 |
| 动态信息 | `<system-reminder>` 追加在消息中 | 不改系统提示词 |
| 切换模型 | 不要中途切换，用子 Agent | 缓存按模型隔离 |
| 工具变更 | 永远不增减工具 | 工具是前缀一部分 |
| 状态切换 | 用工具本身建模 | 工具列表不变 |
| 大量工具 | defer_loading 桩 + ToolSearch | 按需加载 |
| 上下文压缩 | 复用主对话完整前缀 | 分叉复用缓存 |

---

## 容易踩的坑

1. 在系统提示词中放时间戳 → 每次请求缓存失效
2. 工具定义顺序不确定（`dict` 遍历）→ 顺序变了等于前缀变了
3. 动态更新工具参数 → 工具定义变了，缓存失效
4. 压缩时用不同的系统提示词 → 无法复用主对话缓存
5. 忘记预留压缩缓冲区 → 上下文用满后没有空间

> **一句话总结**：把系统提示词和工具定义当作不可变的，把所有变化都推到对话消息的末尾。
