# Anthropic Prompt Caching 实战教程

> 本文基于 Claude Code 团队的工程实践总结，帮助你理解 Prompt 缓存的原理并在自己的 Agent 项目中正确使用它。

---

## 1. 什么是 Prompt Caching

每次调用 Claude API 时，模型需要处理你发送的所有 token（系统提示词、工具定义、对话历史等）。随着对话变长，这个成本会越来越高。

Prompt 缓存允许 API **复用之前请求中已经计算过的部分**，从而大幅降低延迟和费用。

简单理解：如果你连续发了两个请求，前面 90% 的内容完全一样，那这 90% 就可以直接命中缓存，只需要计算最后 10% 的新内容。

---

## 2. 核心原理：前缀匹配

Prompt 缓存的匹配方式是**严格的前缀匹配**（Prefix Matching）：

```
请求内容:  [A] [B] [C] [D] [E（新）]
缓存内容:  [A] [B] [C] [D]
                              ↑ 从这里开始才需要计算
```

关键规则：**从第一个字节开始逐字匹配，一旦某处不同，该处之后的所有内容全部缓存失效。**

```
请求:  [A] [B'] [C] [D] [E]     ← B 变了
缓存:  [A] [B]  [C] [D]
             ↑ 从这里开始全部失效，C、D 也无法复用
```

这意味着：**改前面一个字，后面全部白算。**

---

## 3. 请求结构的最佳排列

既然缓存是前缀匹配，那么内容的排列顺序就至关重要。原则是：**越稳定的内容越靠前，越容易变的内容越靠后。**

Claude Code 的请求结构如下：

```
┌─────────────────────────────────┐
│  ① 静态系统提示词 & 工具定义     │  ← 全局不变，所有会话共享缓存
│     cache_control: ephemeral     │
├─────────────────────────────────┤
│  ② 项目配置（如 CLAUDE.md）      │  ← 项目级不变，同项目会话共享缓存
│     cache_control: ephemeral     │
├─────────────────────────────────┤
│  ③ 对话历史消息                  │  ← 每轮追加，之前的内容不变
│     （中间消息无需标记断点）       │
├─────────────────────────────────┤
│  ④ 最新一轮消息                  │  ← 仅此处为新内容
│     cache_control: ephemeral     │
└─────────────────────────────────┘
```

---

## 4. `cache_control` 断点怎么打

### 4.1 不是每条消息都需要标记

`cache_control: {"type": "ephemeral"}` 是告诉 API "在这里设一个缓存断点"。但你**不需要给每条消息都打标记**。

因为缓存是前缀匹配，中间的消息只要前面的内容没变，就会**自动被包含在缓存前缀中**。

### 4.2 实际的标记策略：滑动窗口

以下是 Claude Code 真实请求中的标记方式：

```json
{
  "system": [
    {
      "text": "You are Claude Code...",
      "cache_control": { "type": "ephemeral" }    // ✅ 断点 1
    },
    {
      "text": "详细的系统指令和环境信息...",
      "cache_control": { "type": "ephemeral" }    // ✅ 断点 2
    }
  ],
  "messages": [
    { "role": "user", "content": "消息1" },         // 无标记
    { "role": "assistant", "content": "回复1" },     // 无标记
    { "role": "user", "content": "消息2" },          // 无标记
    { "role": "assistant", "content": "回复2" },     // 无标记
    // ... 中间可能有很多轮对话，都没有标记 ...
    {
      "role": "assistant",
      "content": [{ "type": "tool_use", "name": "AskUser", "input": {...},
        "cache_control": { "type": "ephemeral" }  // ✅ 断点 3（倒数第二条）
      }]
    },
    {
      "role": "user",
      "content": [{
        "type": "tool_result", "content": "用户的回答",
        "cache_control": { "type": "ephemeral" }  // ✅ 断点 4（最后一条）
      }]
    }
  ]
}
```

### 4.3 为什么这样标记

每一轮新请求，断点会随着对话"向后滑动"：

```
第 N 轮:    [system ✅] [...历史...] [消息N-1 ✅] [消息N ✅]
第 N+1 轮:  [system ✅] [...历史..., 消息N-1, 消息N] [消息N+1 ✅] [消息N+2 ✅]
                                                       ↑ 断点滑到这里了
```

上一轮末尾的消息，在下一轮变成了历史的一部分，自然被前缀缓存覆盖。新的断点只标在最新的末尾。这样每轮只有最新的少量 token 需要计算。

## 5. 动态信息怎么处理：用消息代替改提示词

### 5.1 问题

假设你需要让模型知道当前时间。如果你把时间写在系统提示词里：

```
第 1 轮 System: "...当前时间是 14:00..."
第 2 轮 System: "...当前时间是 14:01..."  ← 改了一个字符
```

系统提示词变了 → 后面所有内容缓存全部失效 → 可能浪费 10 万+ token 的计算。

### 5.2 解决方案

系统提示词保持不变，通过**在对话消息中追加提醒**来传递更新：

```
System: "你是一个助手。"               ← 永远不变

Messages: [
  ...,
  {
    "role": "user",
    "content": "<system-reminder>现在是 14:01</system-reminder>\n用户的实际问题"
  }
]
```

Claude Code 就是这样做的——在用户消息中嵌入 `<system-reminder>` 标签，传递日期、可用技能列表、项目配置（CLAUDE.md）等动态信息。只有信息确实发生变化时才插入，开销仅为每次多几十个 token。

**类比**：这就像数据库中的 append-only log 模式——不去修改原始记录（代价大），而是追加变更日志（代价小）。

## 6. 绝对不要做的事情

### 6.1 不要中途切换模型

Prompt 缓存按模型隔离。如果你在 Opus 的对话中已有 10 万 token，切换到 Haiku 需要为 Haiku **从零重建缓存**，反而更贵。

正确做法：使用**子 Agent 模式**。主模型准备一个精简的上下文交接给子模型，而不是让子模型继承整个对话。

### 6.2 不要中途增减工具

工具定义是缓存前缀的一部分。增加或移除任何一个工具，等于改了前缀，缓存全部失效。

**正确做法 1：用工具模拟状态切换**

Claude Code 的 Plan 模式没有替换工具集，而是设计了 `EnterPlanMode` / `ExitPlanMode` 作为工具本身。工具列表永远不变，通过消息告知模型"你现在处于 Plan 模式，不要编辑文件"。

**正确做法 2：延迟加载（defer_loading）**

对于大量 MCP 工具，不全部加载也不移除，而是发送轻量级的桩（stub）：

```json
{ "name": "some_mcp_tool", "defer_loading": true }
```

模型通过 `ToolSearch` 工具按需发现和加载完整定义。桩始终存在且顺序不变，缓存前缀保持稳定。

## 7. 上下文压缩（Compaction）也要保缓存

当对话超出上下文窗口时，需要压缩（总结对话后继续新会话）。

### 7.1 错误做法

用一个新的 API 请求、不同的系统提示词、不带工具来生成摘要：

```
主对话:  [System A] [Tools] [Messages...]     ← 有缓存
压缩请求: [System B] [Messages...]             ← 前缀完全不同，缓存全部失效
```

所有 token 按全价计算，成本暴增。

### 7.2 正确做法：缓存安全的分叉

压缩请求使用与主对话**完全相同**的系统提示词、工具定义和历史消息，只在末尾追加压缩指令：

```
压缩请求: [System A] [Tools] [Messages...] [压缩指令]
                                             ↑ 只有这里是新的，前面全部命中缓存
```

## 8. 总结速查表

| 原则 | 做法 | 原因 |
|------|------|------|
| 内容排列 | 静态在前，动态在后 | 前缀匹配，前面不变才能命中缓存 |
| 缓存断点 | 只标在 system prompt 和对话末尾 | 滑动窗口策略，中间消息自动被覆盖 |
| 动态信息 | 用 `<system-reminder>` 追加在消息中 | 不改系统提示词，保住前缀缓存 |
| 切换模型 | 不要中途切换，用子 Agent | 缓存按模型隔离，切换等于从零开始 |
| 工具变更 | 永远不增减工具 | 工具是前缀的一部分，改了全部失效 |
| 状态切换 | 用工具本身建模（如 EnterPlanMode） | 工具列表不变，通过消息传递状态 |
| 大量工具 | defer_loading 轻量桩 + ToolSearch | 桩固定存在，按需加载完整定义 |
| 上下文压缩 | 复用主对话的完整前缀 | 分叉请求也能命中父会话缓存 |
| 监控 | 像监控可用性一样监控缓存命中率 | 几个百分点的下降就会显著影响成本 |

---

## 附录：容易踩的坑

1. **在系统提示词中放时间戳** → 每秒都在变，每次请求都缓存失效
2. **工具定义顺序不确定**（如用了 `dict` 遍历）→ 顺序变了等于前缀变了
3. **动态更新工具参数**（如可调用的 Agent 列表）→ 工具定义变了，缓存失效
4. **压缩时用不同的系统提示词** → 无法复用主对话缓存
5. **忘记预留压缩缓冲区** → 上下文用满后没有空间放压缩指令和输出

> **一句话总结**：把系统提示词和工具定义当作不可变的（immutable），把所有变化都推到对话消息的末尾。缓存从头往后匹配，前面不动，后面追加，这就是 Prompt 缓存的全部秘诀。
