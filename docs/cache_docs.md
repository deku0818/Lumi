# 构建 Claude Code 的经验：Prompt 缓存就是一切

> 工程界有句名言："缓存主宰一切"（Cache Rules Everything Around Me），对于 AI Agent 同样适用。

像 Claude Code 这样的长时间运行的 Agent 产品，之所以可行，核心依赖于 **Prompt 缓存**——它允许我们复用之前请求的计算结果，从而显著降低延迟和成本。

在 Claude Code 中，我们围绕 Prompt 缓存构建了整个系统架构。高缓存命中率能降低成本并支撑更慷慨的订阅计划速率限制，因此我们对缓存命中率设置了监控告警，如果命中率过低，会直接触发事故响应（SEV）。

以下是我们在大规模优化 Prompt 缓存过程中总结的经验教训（很多是反直觉的）。

---

## 一、合理排列 Prompt 结构以优化缓存

Prompt 缓存基于**前缀匹配**——API 会从请求开头一直缓存到每个 `cache_control` 断点。这意味着内容的排列顺序至关重要，你需要让尽可能多的请求共享同一前缀。

最佳实践是**静态内容在前，动态内容在后**。在 Claude Code 中的排列如下：

1. **静态系统提示词 & 工具定义**（全局缓存）
2. **Claude.MD 配置文件**（项目级缓存）
3. **会话上下文**（会话级缓存）
4. **对话消息**

这样可以最大化跨会话的缓存命中。

但这个顺序**非常脆弱**！我们曾因以下原因意外打破缓存：

- 在静态系统提示词中放入了详细的时间戳
- 非确定性地打乱了工具定义的顺序
- 更新了工具参数（例如 AgentTool 可调用的子 Agent 列表）

---

## 二、通过消息传递更新信息

当 Prompt 中的信息过时（如时间变化、用户修改了文件），你可能会想直接更新 Prompt，但这会导致**缓存失效**，代价昂贵。

更好的做法是：**通过下一轮对话的消息传入更新信息**。在 Claude Code 中，我们在下一条用户消息或工具返回结果中添加 `<system-reminder>` 标签来传递更新（如"现在是星期三"），从而保住缓存。

---

## 三、不要在会话中途切换模型

Prompt 缓存是**按模型独立**的，这使得缓存的成本计算非常反直觉。

例如：如果你在与 Opus 的对话中已有 10 万 token，想问一个简单问题，切换到 Haiku 反而**更贵**——因为需要为 Haiku 重建整个缓存。

如果确实需要切换模型，最佳方式是使用**子 Agent**：由 Opus 准备一个"交接消息"发送给另一个模型来完成任务。Claude Code 中的 Explore Agent 就是这样使用 Haiku 的。

---

## 四、不要在会话中途增减工具

在对话中途改变工具集是破坏缓存最常见的方式之一。虽然"只给模型当前需要的工具"看似合理，但因为工具定义是缓存前缀的一部分，增减任何工具都会**使整个对话的缓存失效**。

---

## 五、Plan 模式——围绕缓存来设计功能

Plan 模式是围绕缓存约束设计功能的典范。

直觉做法是：用户进入 Plan 模式时，将工具集替换为只读工具。但这会破坏缓存。

我们的做法是：**始终保留所有工具**，将 `EnterPlanMode` 和 `ExitPlanMode` 设计为工具本身。当用户开启 Plan 模式时，Agent 收到一条系统消息说明当前处于 Plan 模式及相应规则——探索代码库、不编辑文件、完成后调用 `ExitPlanMode`。工具定义始终不变。

额外收益：因为 `EnterPlanMode` 是一个模型可以自行调用的工具，它可以在检测到复杂问题时**自主进入 Plan 模式**，且不会破坏缓存。

---

## 六、工具搜索——延迟加载而非移除

同样的原则适用于工具搜索功能。Claude Code 可能加载了数十个 MCP 工具，全部包含在每个请求中会很昂贵，但中途移除会破坏缓存。

解决方案：**延迟加载（defer_loading）**。我们不移除工具，而是发送轻量级的桩（stub）——只有工具名称和 `defer_loading: true` 标记。模型可以通过 `ToolSearch` 工具按需"发现"它们，完整的工具 schema 只在模型选择时才加载。这样缓存前缀保持稳定：相同的桩始终以相同顺序存在。

---

## 七、上下文压缩（Compaction）——分叉时保持缓存

当上下文窗口用尽时，需要对对话进行总结压缩并在新会话中继续。这个过程存在许多与缓存相关的边界情况。

### 问题

简单实现是：用不同的系统提示词和无工具定义发起单独的 API 调用来生成摘要。但这与主对话的缓存前缀完全不匹配，所有输入 token 都要全价计算，成本剧增。

### 解决方案——缓存安全的分叉

压缩时，我们使用与父对话**完全相同**的系统提示词、用户上下文、系统上下文和工具定义。先放入父对话的消息，然后在末尾追加压缩指令作为新的用户消息。

从 API 角度看，这个请求与父对话的最后一次请求几乎一致——相同的前缀、工具和历史——因此缓存前缀可以复用，唯一新增的 token 只是压缩指令本身。

需要注意的是，我们需要预留一个"压缩缓冲区"，确保上下文窗口中有足够空间容纳压缩消息和摘要输出。

---

## 核心经验总结

| 原则 | 说明 |
|------|------|
| **缓存是前缀匹配** | 前缀中任何位置的改动都会使其后所有内容的缓存失效。围绕这个约束设计整个系统。 |
| **用消息代替修改系统提示词** | 进入 Plan 模式、更新日期等操作应插入对话消息中，而非修改系统提示词。 |
| **不要中途切换工具或模型** | 用工具来建模状态转换（如 Plan 模式），用延迟加载代替移除工具。 |
| **像监控可用性一样监控缓存命中率** | 缓存命中率下降几个百分点就会严重影响成本和延迟，应设告警并作为事故处理。 |
| **分叉操作需共享父级前缀** | 压缩、摘要等侧计算应使用与主对话相同的缓存参数，以命中父级缓存。 |

> **结论**：Claude Code 从第一天起就围绕 Prompt 缓存构建，如果你正在构建 Agent，你也应该这样做。
