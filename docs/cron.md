# OpenClaw Cron 机制深度解析

> 本文档详细介绍 OpenClaw 项目中的 Cron（定时任务）子系统的设计原理、核心特性与架构细节，适合用于理解和复刻学习。

---

## 一、整体架构概览

OpenClaw 的 Cron 系统是一个**内嵌于 Gateway 的持久化任务调度引擎**，而非独立的外部调度器。它的核心职责是：按照用户定义的时间规则，周期性或一次性地触发 Agent 行为，并将结果投递到指定的消息通道。

整个系统可以拆分为 **五大子系统**：

```
┌─────────────────────────────────────────────────────────┐
│                      Gateway 进程                        │
│                                                         │
│  ┌───────────┐   ┌───────────┐   ┌──────────────────┐  │
│  │  调度引擎  │──▶│  执行引擎  │──▶│  投递引擎        │  │
│  │ (Timer)   │   │ (Runner)  │   │ (Delivery)       │  │
│  └─────┬─────┘   └─────┬─────┘   └──────────────────┘  │
│        │               │                                │
│  ┌─────▼─────┐   ┌─────▼─────┐   ┌──────────────────┐  │
│  │ 持久化存储 │   │ 容错与重试 │   │ 遥测与日志       │  │
│  │ (Store)   │   │ (Retry)   │   │ (RunLog/Events)  │  │
│  └───────────┘   └───────────┘   └──────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**关键设计原则：**

- **内存优先，磁盘持久化**：所有任务在启动时加载到内存，运行时在内存中操作，变更后原子写回磁盘
- **单进程调度**：不依赖外部调度器（如 systemd timer、外部 cron），由 Gateway 进程内的定时器驱动
- **双轨执行模型**：支持"主会话事件注入"和"隔离 Agent 独立执行"两种完全不同的执行路径

## 二、任务定义模型

一个完整的 Cron Job 由四个核心维度定义：

### 2.1 调度规则（Schedule）

系统支持三种调度类型：

| 类型 | 含义 | 典型场景 |
|------|------|----------|
| **at** | 绝对时间点，一次性触发 | "20 分钟后提醒我"、"明天上午 9 点执行" |
| **every** | 固定间隔循环 | "每 30 分钟检查一次"、"每 2 小时同步" |
| **cron** | 标准 cron 表达式（5/6 字段） | "每周一 9:00"、"每天凌晨 3 点" |

**cron 表达式**使用 `croner` 库解析，支持标准的 5 字段格式（分 时 日 月 周）和 6 字段格式（含秒），并且支持 IANA 时区字符串（如 `Asia/Shanghai`）。

**at 类型**接受 ISO 8601 时间戳，触发后自动禁用或删除（取决于配置）。

**every 类型**基于锚点时间计算下次执行：`下次执行 = 锚点 + ⌈(当前时间 - 锚点) / 间隔⌉ × 间隔`。锚点默认为任务创建时间，也可手动指定。

### 2.2 执行载荷（Payload）

定义"执行什么"，分为两种：

| 类型 | 执行方式 | 上下文 |
|------|----------|--------|
| **systemEvent** | 向主会话注入一条系统事件文本 | 共享主会话的完整对话上下文 |
| **agentTurn** | 启动一个隔离的 Agent 回合 | 独立的临时会话，可指定模型和思考级别 |

### 2.3 投递目标（Delivery）

定义"结果发到哪里"，三种模式：

| 模式 | 行为 |
|------|------|
| **announce** | 通过消息通道（Slack、Telegram、WhatsApp 等）发送结果 |
| **webhook** | HTTP POST 到指定 URL，携带 Bearer Token |
| **none** | 不投递，仅记录日志 |

### 2.4 执行目标（Session Target）

| 目标 | 含义 |
|------|------|
| **main** | 在主 Agent 会话中执行，与用户正在进行的对话共享上下文 |
| **isolated** | 创建独立的临时会话执行，互不干扰 |

## 三、调度引擎工作原理

### 3.1 定时器循环

调度引擎的核心是一个**单定时器循环**，而非每个任务一个定时器：

```
启动 → 加载所有任务 → 计算每个任务的 nextRunAtMs
  ↓
设置定时器 → 等待到最近一个任务的触发时间
  ↓
定时器触发 → 扫描所有任务 → 找出所有到期任务
  ↓
按顺序/并发执行到期任务 → 更新状态 → 持久化
  ↓
重新计算下次触发时间 → 重新设置定时器 → 回到等待
```

**关键细节：**

- **最大定时器延迟**：60 秒。即使下一个任务在 1 小时后，定时器也最多睡 60 秒就醒来检查一次。这是为了防止系统时钟漂移或任务动态添加导致错过执行
- **并发控制**：通过 `maxConcurrentRuns` 配置并行执行的任务数量，默认为 1（串行执行）
- **到期判定**：`isJobDue(job, nowMs)` —— 当前时间 >= 任务的 nextRunAtMs 即为到期

### 3.2 自动错峰（Stagger）机制

这是一个非常精巧的设计，解决了"整点雪崩"问题。

**问题场景**：多个任务都设置了 `0 * * * *`（每小时整点），会在同一时刻同时触发，造成瞬时负载尖峰。

**解决方案**：

1. **自动检测**：识别"每小时整点"模式（分钟字段为 0，小时字段为通配符 `*`）
2. **确定性偏移**：对任务 ID 做 SHA256 哈希，取前 4 字节对错峰窗口取模，得到一个 0～300 秒（5 分钟）的偏移量
3. **应用偏移**：将偏移量加到计算出的下次执行时间上

**关键特点**：

- 偏移量是**确定性的**：同一个任务 ID 每次计算得到的偏移量相同，不会在每次重启后变化
- **只对通配小时生效**：固定时间点的任务（如 `0 7 * * *` 每天 7 点）不会被错峰，因为用户设置了精确时间说明有明确意图
- **可手动覆盖**：设置 `staggerMs = 0` 可禁用错峰，要求精确执行

### 3.3 cron 表达式解析优化

- **LRU 缓存**：最多缓存 512 个已解析的 cron 表达式，避免重复解析开销。缓存键为 `{时区}\0{表达式}`
- **时区回滚 Bug 修复**：`croner` 库在某些时区（如 `Asia/Shanghai`）的年份切换时会返回过去的时间戳。系统会检测这种情况，先从下一秒重试，再从明天 UTC 重试

## 四、双轨执行模型

这是整个 Cron 系统最核心的架构决策。

### 4.1 主会话路径（Main Session）

```
Cron 触发
  → 将文本注入为系统事件（System Event）
  → 根据 wakeMode 决定何时唤醒
     ├── "now"：立即请求心跳（Heartbeat）
     └── "next-heartbeat"：等待下一次常规心跳
  → 心跳处理时读取系统事件
  → Agent 在主会话上下文中处理该事件
```

**适用场景**：需要主会话上下文的任务，比如"检查一下我们刚才讨论的那个问题的进展"。

**wakeMode 的区别**：
- `now`：立刻唤醒 Agent 处理，适合时间敏感的任务
- `next-heartbeat`：事件排队等待，适合不紧急的定期提醒

### 4.2 隔离执行路径（Isolated Agent）

```
Cron 触发
  → 创建临时会话（Key 格式：<agentId>:cron:<jobId>:run:<uuid>）
  → 启动独立的 Agent 回合
  → Agent 执行完毕，提取输出文本
  → 解析投递目标（通道 + 收件人）
  → 通过 announce/webhook 投递结果
  → 记录遥测数据
  → 临时会话在保留期后被清理
```

**适用场景**：独立的定时任务，如"每天早上 9 点总结新闻"、"每小时检查系统状态并发送报告到 Slack"。

**会话生命周期**：
- 每次执行创建一个新的临时会话（UUID 唯一标识）
- 默认保留 24 小时后由 Session Reaper 清理
- 保留期可配置（如 `7d`、`1h30m`），也可设为 `false` 禁用清理
- Reaper 每 5 分钟扫描一次，避免频繁 IO

## 五、容错与重试机制

### 5.1 瞬态错误识别

系统通过正则匹配自动识别四类瞬态错误：

| 错误类别 | 匹配特征 |
|----------|----------|
| **限流** | rate limit、429、too many requests、resource exhausted |
| **网络** | network、econnreset、econnrefused、fetch failed、socket |
| **超时** | timeout、etimedout |
| **服务端错误** | 5xx 状态码 |

### 5.2 退避重试策略

对一次性任务（at 类型），遇到瞬态错误时自动重试：

```
第 1 次失败 → 等待 30 秒后重试
第 2 次失败 → 等待 60 秒后重试
第 3 次失败 → 等待 5 分钟后重试
第 4 次失败 → 等待 15 分钟后重试
第 5 次及以后 → 等待 60 分钟后重试
```

- 默认最多重试 3 次（可配置）
- 连续错误计数器在成功后归零
- 退避时间表完全可配置

### 5.3 任务超时保护

| 任务类型 | 默认超时 |
|----------|----------|
| 系统事件（main session） | 10 分钟 |
| 隔离 Agent 回合 | 60 分钟 |

通过 `Promise.race()` + `AbortController` 实现，超时后自动中止并清理资源。隔离 Agent 任务还支持在载荷中指定自定义超时。

### 5.4 调度错误自动禁用

如果一个任务的调度计算反复出错（比如无效的 cron 表达式导致无法计算下次执行时间），系统会追踪 `scheduleErrorCount`，超过阈值后自动禁用该任务，防止紧密循环消耗资源。

### 5.5 失败告警

支持全局或单任务级别的失败告警配置：

- **触发条件**：连续失败 N 次后触发（默认可配置）
- **告警通道**：可以发送到与任务输出不同的通道（比如任务结果发 Slack，失败告警发 Telegram）
- **冷却期**：默认 1 小时内不重复告警，避免告警风暴

## 六、持久化机制

### 6.1 存储格式

任务存储在 `~/.openclaw/cron/jobs.json`，使用 JSON5 格式（支持注释和尾逗号），结构为：

```
{
  version: 1,
  jobs: [ ...所有任务定义及其运行时状态... ]
}
```

每个任务不仅包含定义信息，还包含运行时状态（`state` 字段）：下次执行时间、上次执行结果、连续错误计数等。

### 6.2 原子写入

为防止写入过程中崩溃导致数据损坏：

1. 先写入临时文件
2. 通过原子重命名（`rename`）替换原文件
3. Windows 上对 `EBUSY` 错误有重试逻辑
4. 写入前创建备份

### 6.3 变更检测

内存中维护序列化缓存，只有当任务数据真正发生变化时才写磁盘，避免不必要的 IO。

### 6.4 启动恢复

Gateway 启动时：

1. 从磁盘加载 `jobs.json`（不存在则创建空文件）
2. 对所有任务重新计算 `nextRunAtMs`
3. 丢弃调度计算出错的任务（记录日志）
4. 设置定时器，开始调度循环

### 6.5 读操作安全性

一个值得注意的细节：读操作（如 `list`）使用**维护模式重计算**，只更新下次执行时间，不会推进已过期任务的状态。这避免了一个微妙的 Bug——如果读操作把过期任务标记为"已跳过"，那么这些任务就永远不会被真正执行。

## 七、执行日志与遥测

### 7.1 Run Log

每个任务有独立的执行日志文件：`~/.openclaw/cron/runs/<jobId>.jsonl`，JSONL 格式（每行一条 JSON 记录）。

每条记录包含：

- 时间戳、状态（ok/error/skipped）、错误信息
- 执行时长、会话 ID
- 投递状态（delivered/not-delivered/unknown/not-requested）
- AI 模型信息（模型名、Provider）
- Token 使用量（输入/输出/缓存读写）

**日志裁剪**：默认最大 2MB 或 2000 行，超出后通过二分查找截断，保持 JSONL 格式完整性。

### 7.2 事件遥测

每个任务生命周期事件（added/updated/removed/started/finished）都通过 `onEvent` 回调发出，供 Gateway 层统一处理遥测上报。

## 八、接口层

系统提供三个层级的接口：

### 8.1 CLI 命令行

```
openclaw cron status          # 查看 Cron 服务状态
openclaw cron list            # 列出所有任务
openclaw cron add             # 创建任务
openclaw cron edit <id>       # 编辑任务
openclaw cron remove <id>     # 删除任务
openclaw cron run <id>        # 立即执行
openclaw cron runs <id>       # 查看执行历史
openclaw cron wake            # 手动唤醒
```

CLI 提供了大量快捷参数，如 `--every 30m`、`--cron "0 9 * * MON"`、`--announce`、`--channel slack` 等，降低使用门槛。

### 8.2 Gateway RPC API

通过 Gateway 的 RPC 协议暴露方法（`cron.status`、`cron.list`、`cron.add` 等），使用 TypeBox Schema 做输入校验。这是 Web UI 和其他客户端的访问入口。

### 8.3 Agent Tool

Agent 在对话中可以通过 Cron Tool 直接操作定时任务。例如用户说"每天早上 9 点给我发天气预报"，Agent 可以调用 Cron Tool 的 `add` 动作来创建任务。Tool 还支持注入最近的聊天上下文到唤醒消息中。

## 九、与其他子系统的集成关系

```
                    ┌──────────────┐
                    │   用户 / CLI  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Gateway RPC │
                    └──────┬───────┘
                           │
              ┌────────────▼────────────┐
              │       CronService       │
              └────┬──────────────┬─────┘
                   │              │
         ┌─────────▼──┐    ┌─────▼──────────┐
         │ 主会话路径   │    │ 隔离执行路径     │
         │            │    │                │
         │ Heartbeat  │    │ Agent Runner   │
         │ System     │    │ Model Fallback │
         │ Events     │    │ Session Mgmt   │
         └─────┬──────┘    └────┬───────────┘
               │                │
               │         ┌──────▼──────┐
               │         │ 投递引擎     │
               │         │             │
               │         │ ├ Announce  │──▶ Slack / Telegram / WhatsApp ...
               │         │ ├ Webhook   │──▶ HTTP POST
               │         │ └ None      │
               │         └─────────────┘
               │
               ▼
        主会话 Agent 处理
```

**关键集成点：**

- **Heartbeat（心跳）系统**：主会话任务通过心跳机制唤醒 Agent
- **Agent 执行层**：隔离任务复用 Agent 的模型回退链、认证传播等基础设施
- **消息通道层**：投递引擎复用所有已注册的消息通道插件（包括扩展）
- **配置系统**：从 `.openclaw.json` 加载 Cron 配置，与全局配置统一管理
- **会话管理**：隔离任务的临时会话纳入统一的会话存储和生命周期管理

## 十、设计亮点总结

| 特性 | 设计思路 |
|------|----------|
| **单定时器 + 全量扫描** | 避免大量定时器的内存开销和管理复杂度 |
| **确定性错峰** | SHA256 哈希确保偏移量稳定，重启后行为一致 |
| **双轨执行** | 灵活适配"上下文相关"和"独立执行"两类需求 |
| **原子持久化** | 临时文件 + 重命名，防止崩溃导致数据损坏 |
| **读操作不推进状态** | 避免"列表查询导致任务被跳过"的微妙 Bug |
| **瞬态错误自动识别** | 正则匹配 + 指数退避，对一次性任务尤其重要 |
| **会话 Reaper** | 自动清理隔离会话，防止存储无限增长 |
| **日志二分裁剪** | 在大小限制内保持 JSONL 格式完整性 |
| **三层接口** | CLI / RPC / Agent Tool 覆盖人工和自动化场景 |
