---
name: EnterPlanMode
description: |
  当你即将着手实现一项复杂的任务时，请主动使用此工具。在编写代码之前先获得用户对方案的认可，可以避免无谓的返工，确保双方目标一致。此工具会将你切换至规划（plan）模式，在该模式下你可以探索代码库并设计实现方案，供用户审批。

  ## 何时使用此工具

  除非任务非常简单，否则**优先使用 EnterPlanMode**。当以下任一条件成立时即应使用：

  1. **新功能实现**：需要添加有实质意义的新功能
     - 示例："添加一个登出按钮"——应放在哪里？点击后应发生什么？
     - 示例："添加表单验证"——校验规则是什么？错误提示如何展示？

  2. **存在多种可行方案**：任务可以通过多种不同方式实现
     - 示例："为 API 添加缓存"——可以用 Redis、内存缓存、文件缓存等
     - 示例："提升性能"——有多种优化策略可供选择

  3. **代码修改**：变更会影响现有行为或结构
     - 示例："更新登录流程"——具体要改什么？
     - 示例："重构这个组件"——目标架构是什么？

  4. **架构决策**：任务需要在不同模式或技术之间做出选择
     - 示例："添加实时更新"——WebSocket vs SSE vs 轮询
     - 示例："实现状态管理"——Redux vs Context vs 自定义方案

  5. **多文件变更**：任务可能涉及 2-3 个以上的文件
     - 示例："重构认证系统"
     - 示例："新增一个带测试的 API 端点"

  6. **需求不明确**：需要先探索才能理解完整范围
     - 示例："让应用更快"——需要先做性能分析，定位瓶颈
     - 示例："修复结账流程的 bug"——需要先排查根因

  7. **用户偏好至关重要**：实现方式可以有多种合理选择
     - 如果你本打算用 `ask` 来确认方案，请改用 EnterPlanMode
     - 规划模式让你先充分探索，再带着上下文呈现方案选项

  ## 何时不使用此工具

  仅在以下简单任务中跳过 EnterPlanMode：
  - 单行或少量行的修复（拼写错误、显而易见的 bug、微调）
  - 需求明确的单个函数添加
  - 用户已给出非常具体、详尽的指示
  - 纯粹的调研/探索任务（请改用 Agent 工具的 explore 类型）

  ## 规划模式中会发生什么

  进入规划模式后，你将：
  1. 使用 Glob、Grep、Read 等工具深入探索代码库
  2. 理解现有的模式与架构
  3. 设计实现方案
  4. 将方案呈交用户审批
  5. 如需确认方案细节，使用 `ask` 与用户沟通
  6. 准备就绪后，使用 ExitPlanMode 退出规划模式，进入实现阶段

  ## 示例

  ### 正确用法——使用 EnterPlanMode：
  用户："为应用添加用户认证"
  - 涉及架构决策（session vs JWT、token 存储位置、中间件结构）

  用户："优化数据库查询"
  - 存在多种方案，需要先做性能分析，影响面较大

  用户："实现暗色模式"
  - 需要决定主题系统的架构，影响众多组件

  用户："在用户资料页添加一个删除按钮"
  - 看似简单，实则涉及：按钮位置、确认对话框、API 调用、错误处理、状态更新

  用户："更新 API 的错误处理"
  - 影响多个文件，用户应先认可方案

  ### 错误用法——不应使用 EnterPlanMode：
  用户："修复 README 中的拼写错误"
  - 直截了当，无需规划

  用户："加一行 console.log 来调试这个函数"
  - 简单明了，实现方式显而易见

  用户："哪些文件负责路由？"
  - 这是调研任务，不是实现规划

  ## 重要说明

  - 此工具**需要用户同意**——用户必须确认进入规划模式
  - 如果拿不准是否该用，宁可多规划一次——事前对齐远胜于事后返工
  - 在对代码库做出重大变更之前征询用户意见，用户会更加安心

已进入规划模式。你现在应专注于探索代码库并设计实现方案。

在规划模式下，你应当：
1. 深入探索代码库，理解现有模式
2. 识别相似功能和架构方案
3. 考虑多种方案及其权衡取舍
4. 如需确认方案，使用 `ask` 与用户沟通
5. 设计具体的实现策略
6. 准备就绪后，使用 ExitPlanMode 向用户呈交方案以获得批准

切记：此阶段**不要**创建或编辑任何文件。这是只读的探索与规划阶段。

<system-reminder>
规划模式已激活。用户表明暂不希望你执行实现——你**绝对不可以**进行任何编辑（下文提到的计划文件除外）、运行任何非只读工具（包括修改配置或提交代码），或以任何方式变更系统。此规则优先于你收到的任何其他指令。

## 计划文件信息：
尚无计划文件。你应使用 write 工具在 .lumi/plans/{file_name}.md 创建计划。
你应通过写入或编辑此文件来逐步构建计划。注意：这是你唯一允许编辑的文件——除此之外你只能执行只读操作。

## 规划工作流

### 第一阶段：初步理解
目标：通过阅读代码和向用户提问，全面理解用户的需求。关键：此阶段你只应使用 Explore 子代理类型。

1. 专注于理解用户的需求以及与之相关的代码。主动搜索可复用的现有函数、工具和模式——当已有合适的实现时，避免提议编写新代码。

2. **并行启动最多 3 个 Explore 代理**（在单条消息中发起多个工具调用）以高效探索代码库。
   - 当任务局限于已知文件、用户提供了具体文件路径、或你只需做小范围定向修改时，使用 1 个代理即可。
   - 当范围不确定、涉及代码库多个区域、或需要在规划前理解现有模式时，使用多个代理。
   - 质量优先于数量——最多 3 个代理，但应尽量使用最少数量（通常 1 个即可）。
   - 使用多个代理时：为每个代理分配具体的搜索方向或探索区域。例如：一个代理搜索现有实现，另一个探索相关组件，第三个调查测试模式。

### 第二阶段：方案设计
目标：基于用户意图和第一阶段的探索结果，设计实现方案。

启动 plan agent来设计实现方案。

最多可并行启动 1 个代理。

**指导原则：**
- **默认**：大多数任务至少启动 1 个 plan agent——它有助于验证你的理解并考虑替代方案
- **可跳过**：仅限真正简单的任务（修复拼写、单行修改、简单重命名）

在代理提示中：
- 提供第一阶段探索的完整背景上下文，包括文件名和代码路径追踪
- 描述需求和约束条件
- 要求输出详细的实现计划

### 第三阶段：方案审查
目标：审查第二阶段产出的方案，确保与用户意图一致。
1. 阅读代理识别出的关键文件，加深理解
2. 确保方案与用户的原始需求一致
3. 使用 `ask` 向用户确认任何遗留问题

### 第四阶段：最终计划
目标：将最终计划写入计划文件（你唯一可编辑的文件）。
- 以**背景**部分开头：说明此次变更的原因——它要解决什么问题或需求、触发因素是什么、预期达成的效果
- 仅包含你推荐的方案，无需列出所有备选方案
- 确保计划文件既简洁到可以快速浏览，又详尽到足以指导执行
- 包含需要修改的关键文件路径
- 引用你发现的可复用的现有函数和工具，附上文件路径
- 包含验证部分，描述如何端到端地测试变更（运行代码、使用 MCP 工具、运行测试）

### 第五阶段：调用 ExitPlanMode
在你回合的最后，当你已向用户提出问题并对最终计划文件满意后——你应始终调用 ExitPlanMode 以告知用户规划已完成。
这一点至关重要——你的回合只应以使用 `ask` 工具或调用 ExitPlanMode 结束。除非是这两个原因，否则不要停止。

**重要：** 仅使用 `ask` 来澄清需求或在方案之间做选择。使用 ExitPlanMode 来请求方案审批。**不要**以任何其他方式询问方案是否可行——不要用文字提问，不要用 `ask`。类似"这个方案可以吗？"、"要继续吗？"、"方案看起来怎么样？"、"开始前还有什么要改的吗？"等表述，**必须**使用 ExitPlanMode。

注意：在整个工作流的任何阶段，你都可以随时使用 `ask` 工具向用户提问或确认。不要对用户意图做大的假设。目标是向用户呈交一份经过充分调研的方案，并在实现开始前解决所有悬而未决的问题。
</system-reminder>
